# BookService - Мини веб-служба и конвейер обработки запросов

Этот проект представляет собой мини веб-службу на ASP.NET Core, которая демонстрирует принципы построения конвейера обработки HTTP-запросов. В рамках проекта реализованы базовые CRUD-операции для каталога книг, а также сквозные функции (логирование, замер времени, стандартизированная обработка ошибок) с использованием механизма Middleware.

## Предметная область: Каталог Книг

Веб-служба предоставляет API для управления коллекцией книг. Каждая книга имеет следующие атрибуты:

*   `Id` (GUID): Уникальный идентификатор книги.
*   `Title` (string): Название книги.
*   `Author` (string): Автор книги.
*   `PublicationYear` (int): Год публикации книги.

### Правила предметной области:

*   `Title` и `Author` являются обязательными полями и не могут быть пустыми.
*   `PublicationYear` должен быть корректным годом (например, между 1000 и текущим годом).

## Описание API и Набор Проверок

Все API-эндпоинты начинаются с `/api/books`.

### 1. Получение списка всех книг

*   **Метод:** `GET`
*   **URL:** `http://localhost:5111/api/books`
*   **Описание:** Возвращает JSON-массив всех книг, хранящихся в памяти.
*   **Ожидаемый результат:** `200 OK` и список книг.

**Команда `curl`:**
```
curl -X GET http://localhost:5111/api/books
```

### Архитектурное пояснение: Аккуратный каркас

В данном проекте реализован подход к построению веб-службы, который обеспечивает чистоту кода, масштабируемость и легкость в сопровождении. Ключевая идея заключается в разделении ответственности и использовании механизма Middleware (промежуточного ПО) в ASP.NET Core.

1. Концепция Конвейера Обработки Запросов (Request Pipeline)
ASP.NET Core обрабатывает каждый входящий HTTP-запрос как последовательность шагов, или "конвейер". Каждый шаг в этом конвейере — это Middleware. Middleware может выполнить какую-то логику, а затем либо передать запрос следующему Middleware, либо сгенерировать ответ и остановить конвейер.

2. Разделение Ответственности (Separation of Concerns)
Вместо того чтобы встраивать логику логирования, замера времени или обработки ошибок непосредственно в каждый контроллер или каждое действие, эти "сквозные" задачи (cross-cutting concerns) вынесены в отдельные Middleware.

Контроллеры (BooksController): Сосредоточены исключительно на бизнес-логике: получение, добавление, валидация данных, специфичная для предметной области "Книги". Они не "знают" о том, как происходит логирование или форматирование ошибок.

Репозиторий (InMemoryBookRepository): Сосредоточен на логике доступа к данным, абстрагируя контроллеры от деталей хранения.

3. Middleware в работе
RequestIdMiddleware

Что делает: для каждого поступающего запроса создаёт уникальный идентификатор (RequestId). Фиксирует в логах старт и завершение обработки запроса, а также указывает HTTP‑метод и URL‑путь.

Польза: даёт возможность сквозного отслеживания запроса. С помощью RequestId можно восстановить полную цепочку действий с запросом в логах — это незаменимо при диагностике проблем и мониторинге в распределённых системах.

TimingMiddleware

Что делает: замеряет, сколько времени ушло на обработку запроса на всех последующих этапах конвейера — в том числе на работу контроллера и репозитория. Записывает результат в логи, привязывая его к RequestId.

Польза: позволяет выявить участки кода, тормозящие работу приложения, и проанализировать производительность системы в целом.

ExceptionHandlerMiddleware

Что делает: ловит все исключения, которые возникают в следующих за ним Middleware либо в бизнес‑логике контроллеров. Преобразует их в стандартизированный JSON‑ответ с описанием ошибки (ErrorResponse), присваивает соответствующий HTTP‑статус (например, 400, 404, 500) и добавляет RequestId.

Польза: гарантирует единообразное представление ошибок для клиентских приложений — это упрощает их обработку на стороне клиента. В продакшн‑среде скрывает детали исключений, предотвращая утечку конфиденциальных данных; полная информация об ошибках выводится только в режиме разработки.

Почему это продуманная архитектура, а не набор разрозненных решений
Модульность и возможность повторного использования. Каждый Middleware — автономный компонент. Его можно без проблем подключить, отключить или перенести в другой проект, не затрагивая бизнес‑логику.

Простота поддержки. Если нужно изменить формат логов или структуру ответов об ошибках, достаточно отредактировать соответствующий Middleware. Не придётся вносить правки во все контроллеры.

Чистота и лаконичность кода. Контроллеры выполняют только свои прямые задачи — их код становится проще для чтения и тестирования. Вся вспомогательная логика (логирование, обработка ошибок и т. д.) вынесена в Middleware.

Гибкость расширения. Чтобы добавить новую сквозную функциональность (аутентификацию, авторизацию, кэширование и т. п.), достаточно внедрить дополнительный Middleware. Существующая логика остаётся нетронутой.

Надёжность системы. Благодаря централизованной обработке ошибок даже неожиданные исключения не приводят к аварийному завершению работы сервиса или выдаче неструктурированных ответов. Клиент всегда получает понятный и стандартизированный ответ.
